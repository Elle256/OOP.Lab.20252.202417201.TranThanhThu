--- Section 11: Polymorphism with toString() ---

Q: Observe and explain what happens when calling m.toString() inside the loop.
A: When calling m.toString() where m is declared as type Media (static type), Java does NOT call Media's toString().
   Instead, it automatically calls the toString() method of the ACTUAL object type at runtime (dynamic type).
    - If m is actually a Book --> Book.toString() is called
    - If m is actually a DigitalVideoDisc --> DigitalVideoDisc.toString() is called
    - If m is actually a CompactDisc --> CompactDisc.toString() is called
This mechanism is called Dynamic Dispatch (late binding), which is the foundation of Polymorphism.
Thanks to this, the same code `m.toString()` can produce different outputs depending on the actual object type.



--- Section 12: Questions about Comparable ---

Q1: What class should implement the Comparable interface?
A: The Media class should implement Comparable<Media>, because all media types (Book, DVD, CD) need comparison capability,
and Media is their common superclass.

Q2: In those classes, how should you implement the compareTo() method be to reflect the ordering that we 
want?
A: In those classes, the compareTo() method should compare objects step by step according to the 
required ordering rule. Suppose we choose the following rule as the natural ordering: Sort by TitleCost
Then the compareTo() method in the Media class can be implemented as follows:

@Override
public int compareTo(Media other) {

    // Step 1:
    // Compare titles alphabetically (A --> Z)
    int titleCmp =
            this.getTitle()
                .compareTo(other.getTitle());

    // If titles are different, return the result immediately
    if (titleCmp != 0) {
        return titleCmp;
    }

    // Step 2:
    // If titles are the same, compare costs in descending order
    return Float.compare(
            other.getCost(),
            this.getCost()
    );
}

Explanation:

The comparison works in two stages:
1. Compare titles
   - compareTo() on String compares alphabetically.

2. If titles are equal
   - compare the costs.
   - Higher cost should come first, so the comparison order is reversed:

         Float.compare(other.getCost(), this.getCost())

   instead of:

         Float.compare(this.getCost(), other.getCost())

Example:

    Java      20
    Java      10
    Batman    15

After sorting:

    Batman    15
    Java      20
    Java      10

This implementation correctly reflects the required ordering rule: title ascending, then cost descending.

Q3: Can we have two ordering rules of the item (by title then cost and by cost then title) if we use this 
Comparable interface approach?
A: NO. Comparable only allows ONE natural ordering through the compareTo() method. To support multiple 
sorting orders, Comparator must be used (as implemented in this lab with COMPARE_BY_TITLE_COST and COMPARE_BY_COST_TITLE).

Q4: Suppose the DVDs has a different ordering rule from the other media types, that is by title, then decreasing 
length, then cost. How would you modify your code to allow this?

A: With Comparable, this is difficult to handle properly, because compareTo() is defined in the superclass.
Although compareTo() could be overridden in DigitalVideoDisc, it may violate the consistency between 
equals() and compareTo() if implemented carelessly. The best solution is to create a separate Comparator
specifically for DVDs:

   Comparator<DigitalVideoDisc> DVD_ORDER =
       Comparator.comparing(DigitalVideoDisc::getTitle)
                 .thenComparing(
                     Comparator.comparingInt(
                         DigitalVideoDisc::getLength
                     ).reversed()
                 )
                 .thenComparing(
                     Comparator.comparingDouble(
                         DigitalVideoDisc::getCost
                     )
                 );
This demonstrates why Comparator is more flexible than Comparable when multiple sorting criteria are needed.
